# Aetheris – Phase 6: Memory Evolution (Update, Merge & Archive)

You are a Senior AI Systems Architect.

We are building **Aetheris**, a modular cognitive AI architecture.

This is **ONLY Phase 6**.

## Strict Scope

Implement **Memory Evolution**.

Do NOT implement:

* Personality
* Emotion
* Reflection
* Planner
* Knowledge Graph
* Tool Calling
* Multi-Agent Systems
* Autonomous Tasks
* Conversation History
* Memory Compression
* Memory Summarization

Only extend the existing Memory System.

---

# Objective

Aetheris should no longer blindly append memories.

Before storing a new memory, it must determine whether it should:

1. Create a new memory
2. Update an existing memory
3. Merge with an existing memory
4. Archive an outdated memory

---

# Architecture

Every new memory follows this pipeline:

User Message

↓

Memory Evaluator

↓

Embedding

↓

Semantic Search (Top 5)

↓

Memory Evolution Engine

↓

Decision

↓

Create / Update / Merge / Archive

↓

Persist Changes

---

# Conflict Detection

Compare the incoming memory with retrieved memories.

Determine if the new memory:

* reinforces an existing fact
* updates an existing fact
* contradicts an existing fact
* is unrelated

---

# Decision Types

The Memory Evolution Engine must return ONLY one of:

CREATE

UPDATE

MERGE

ARCHIVE

# Versioning

Never permanently overwrite a memory.

Every update must:

Increment version.

Store previous value inside history.

Example

Version 1

Favorite language = Java

↓

Version 2

Favorite language = Rust

History

Version 1 stored.

---

# Archive Rules

Archive memories that become obsolete.

Examples

Temporary travel plans

Old preferences replaced by new ones

Completed temporary tasks

Archived memories must NOT appear in normal retrieval.

They remain accessible for historical queries.

---

# ChromaDB

If a memory is updated or merged:

Update both:

* vector representation
* metadata

Never leave stale embeddings.
