# Current Bug

During an active session:

User:
> Don't call me John Paul anymore.
> Call me Creator.

Aetheris correctly follows the instruction.

However, after restarting Aetheris:

it retrieves the old memory

"John Paul"

instead of

"Creator".

This indicates that the update is applied only to the current conversation/session and is **not being permanently committed to long-term memory**.

---

# Goal

When a user changes an existing memory, the change must persist across application restarts.

Nothing else should change.

---

# Scope

Only fix persistent memory updates.

Do not redesign the memory system.

Do not introduce new memory types.

Do not change retrieval ranking.

Do not modify embeddings unless required for consistency.

# Update Rules

The system should recognize update intents such as:

- My name is ...
- Call me ...
- Refer to me as ...
- Don't call me ...
- Change my name to ...
- Update my name to ...

When such an intent is detected:

- Update the existing memory.
- Do not create duplicate active memories.
- Ensure only one active value exists for that memory category.

---

# Persistence

Verify that every successful memory update is written to the persistent storage currently used by Aetheris.

If SQLite is used as the source of truth:

- update SQLite

If ChromaDB stores semantic memories:

- update or regenerate the corresponding vector entry only if required.

Both stores must remain synchronized.

---
