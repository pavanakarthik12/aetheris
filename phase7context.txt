# Aetheris – Phase 9: Hierarchical Memory Architecture

You are a Principal AI Systems Architect.

This is NOT a feature request.

This is a core architectural redesign of the memory hierarchy.

Do NOT redesign unrelated systems.

Do NOT modify the LLM provider layer.

Do NOT remove existing functionality.

Your task is to separate memory into independent layers and ensure the Cognitive Request Router chooses the correct memory source for every query.

---

# Problem

Currently Aetheris treats all memories as one collection.

As a result:

* Old long-term memories leak into simple conversations.
* Current conversation is confused with historical memories.
* Personal memories appear in unrelated questions.
* "What was our previous conversation?" incorrectly searches long-term memory.
* Simple math questions include unrelated user facts.

This architecture is incorrect.

---

# Objective

Separate memory into three completely independent systems.

Each system has a distinct purpose.

The router must decide which one to use.

---

# Memory Layer 1 – Short-Term Conversation Memory

Purpose

Store only the current conversation.

Contains

* user messages
* assistant replies

Capacity

Configurable

Default

20 exchanges

This memory is temporary.

When a new conversation begins,

old conversation history should not automatically be reused.

---

# Short-Term Queries

Examples

"What did I ask before?"

"What was our previous conversation?"

"What was your last answer?"

These queries must search ONLY Short-Term Conversation Memory.

Never search ChromaDB.

---

# Memory Layer 2 – Long-Term User Memory

Purpose

Store persistent user facts.

Examples

* Name
* Preferences
* Favorite Language
* Current Project
* Birthday
* Food Preferences
* Goals

Storage

ChromaDB

This memory persists across sessions.

---

# Long-Term Queries

Examples

"What is my favorite language?"

"What project am I am building?"

"What is my name?"

These queries search ONLY Long-Term Memory.

Never use conversation history.

---

# Memory Layer 3 – System Memory

Purpose

Store information about Aetheris itself.

Examples

* Assistant identity
* Capabilities
* Version
* Available tools
* Personality configuration

This memory must never mix with user memories.

---

# System Queries

Examples

"Who are you?"

"What can you do?"

"What model are you using?"

These queries use ONLY System Memory.

Never retrieve user memories.

---

---

# Context Builder

Build prompts from only the required memory layer.

Do NOT combine all memory sources.

Only merge layers when explicitly required.

---

# Conversation Reset

Implement conversation sessions.

Each session has:

Session ID

Created Time

Last Activity

Conversation History

When a new session starts,

Conversation Memory begins empty.

Long-Term Memory remains unchanged.

---

# Memory Isolation

Conversation Memory

must never be embedded into ChromaDB.

Long-Term Memory

must never automatically appear in unrelated chats.

System Memory

must never be editable by normal user memory commands.

---

# Retrieval Rules

Math

No memories.

Greeting

No memories.

Programming

Relevant conversation only.

Memory Query

Long-Term Memory only.

Conversation Query

Conversation Memory only.

System Query

System Memory only.

# Deliverables

1. Separate memory into Conversation Memory, Long-Term User Memory, and System Memory.
2. Update the Cognitive Request Router to choose the correct memory layer automatically.
3. Prevent unrelated memories from leaking into responses.
4. Ensure "previous conversation" queries search only the active conversation session.
5. Preserve backward compatibility with the existing Memory Service.
6. Produce a memory hierarchy diagram and document the routing logic for each query type.

The final result should make Aetheris behave like a true cognitive system with clearly separated short-term memory, long-term memory, and self-identity, eliminating context pollution and improving response accuracy.
